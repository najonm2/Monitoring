1. System Overview & Architecture

1.1 What Is the PASE Monitor Portal?

The PASE Monitor Portal is a web-based application that consolidates Informatica job monitoring data from multiple Oracle databases into a single, unified dashboard. It provides real-time visibility into job statuses, failures, and performance metrics across four distinct applications.

Key Concept: Instead of manually querying multiple databases and running complex SQL statements, users simply open a web browser, click on an application, and instantly see formatted, up-to-date monitoring data.

2. Technology Stack Explained

2.1 Django Framework - The Foundation

What is Django? Django is a Python web framework that follows the "Model-View-Template" (MVT) pattern. It handles all the complex parts of web development automatically.

Django Provides:

  • URL Routing: Maps URLs like /api/reports/level3/... to Python functions
  • Template Engine: Combines HTML with data to create dynamic pages
  • ORM (Object-Relational Mapping): Works with databases using Python objects (we use for SQLite metadata)
  • Security: Protection against SQL injection, XSS attacks, CSRF attacks
  • Admin Interface: Built-in admin panel at /admin/
Why Django?
  • ✅ Battle-tested by huge companies (Instagram, Pinterest, Mozilla)
  • ✅ "Batteries included" - has everything we need built-in
  • ✅ Excellent documentation and community support
  • ✅ Security features enabled by default
  • ✅ Easy to maintain and scale

3.3 How Connection Works (Technical Details)

Step-by-Step Connection Process:

  1. DNS Resolution: Python resolves hostname (e.g., RACORAP32-SCAN) to IP address
  2. TCP Connection: Opens TCP socket to Oracle server on port 1521
  3. Oracle Protocol Handshake: python-oracledb sends connection request with service name
  4. Authentication: Oracle validates username and password
  5. Session Established: Connection object returned, ready to execute queries

6. Database Query Implementation

6.1 Level3 Failed Jobs with Error - Detailed Breakdown

This is one of the most complex queries. Let's understand it step by step.

The Query:

-- Step 1: Find all workflow runs that had errors (CTE) WITH error_sessions AS ( SELECT DISTINCT workflow_run_id FROM REP_SESS_LOG WHERE instance_name = 'IDG01P' -- Our instance AND end_time >= SYSDATE - 1 -- Last 24 hours AND UPPER(task_name) LIKE '%ERROR%' -- Task name contains "ERROR" ) -- Step 2: Get details for those workflows SELECT w.workflow_name, -- Workflow name s.session_name, -- Session name s.start_time, -- When it started s.end_time, -- When it ended ROUND((s.end_time - s.start_time) * 24 * 60, 2) as duration_minutes, -- Duration LISTAGG(t.task_name, ', ') WITHIN GROUP (ORDER BY t.task_name) as error_message -- All error tasks FROM REP_WFLOW_RUN w -- Workflow run table JOIN REP_SESS_LOG s ON w.workflow_run_id = s.workflow_run_id -- Join sessions JOIN REP_TASK_INST_RUN t ON s.workflow_run_id = t.workflow_run_id -- Join tasks WHERE w.workflow_run_id IN (SELECT workflow_run_id FROM error_sessions) -- Only error workflows AND s.instance_name = 'IDG01P' AND s.end_time >= SYSDATE - 1 GROUP BY w.workflow_name, s.session_name, s.start_time, s.end_time ORDER BY s.start_time DESC -- Most recent first

Query Execution Steps:

  1. CTE (Common Table Expression): Finds all workflow_run_ids that have tasks with "ERROR" in the name
    • Scans REP_SESS_LOG table
    • Filters by instance (IDG01P) and time (last 24 hours)
    • Finds task_names containing "ERROR"
    • Returns distinct workflow_run_ids
  2. Main Query: Joins three tables to get complete information
    • REP_WFLOW_RUN: Workflow execution records
    • REP_SESS_LOG: Session execution logs
    • REP_TASK_INST_RUN: Individual task runs
  3. LISTAGG Function: Aggregates multiple error task names into single comma-separated string
  4. Duration Calculation: Converts timestamp difference to minutes
  5. Filtering: Only includes workflows from error_sessions CTE
  6. Grouping: Groups by workflow and session to aggregate error messages
  7. Sorting: Orders by start_time descending (newest first)

6.2 Level3 Long Running Sessions - Performance Comparison

This query identifies sessions running longer than usual by comparing to historical averages.

SELECT s.workflow_name, s.session_name, s.start_time, s.end_time, ROUND((s.end_time - s.start_time) * 24 * 60, 2) as duration_minutes, -- Calculate 7-day average duration for this session ROUND(AVG(CASE WHEN hist.end_time - hist.start_time > 0 THEN (hist.end_time - hist.start_time) * 24 * 60 ELSE NULL END) OVER (PARTITION BY s.session_name), 2) as avg_duration_minutes, -- Compare current to average ROUND( ((s.end_time - s.start_time) * 24 * 60) / NULLIF(AVG(CASE WHEN hist.end_time - hist.start_time > 0 THEN (hist.end_time - hist.start_time) * 24 * 60 ELSE NULL END) OVER (PARTITION BY s.session_name), 0) * 100, 2) as percent_of_average FROM REP_SESS_LOG s LEFT JOIN REP_SESS_LOG hist ON s.session_name = hist.session_name AND hist.start_time >= SYSDATE - 7 -- Last 7 days AND hist.start_time < SYSDATE - 1 -- Exclude today AND hist.end_time > hist.start_time -- Valid durations only WHERE s.instance_name = 'IDG01P' AND s.start_time >= SYSDATE - 1 -- Today's runs AND s.end_time > s.start_time -- Completed sessions -- Only show sessions running longer than 150% of average HAVING duration_minutes > (avg_duration_minutes * 1.5) ORDER BY percent_of_average DESC -- Worst offenders first

Logic Explanation:

  • Self-Join: Joins REP_SESS_LOG to itself to compare current runs with historical runs
  • Window Function: AVG(...) OVER (PARTITION BY ...) calculates average for each session name
  • 7-Day Window: Only considers runs from past 7 days for average calculation
  • 150% Threshold: Flags sessions running >1.5x their normal time
  • Percentage Calculation: Shows how much longer than average (e.g., 200% = twice as long)

from django.http import JsonResponse from portal.services import level3_service def api_report_data(request, app_id, report_id): """ Main API endpoint handler. Routes requests to appropriate service functions. Args: request: Django HttpRequest object app_id: Application identifier (e.g., 'level3', 'mdm') report_id: Report identifier (e.g., 'lvl3-failed-with-error') Returns: JsonResponse with data or error """ try: data = [] # Route to appropriate service based on app_id and report_id if app_id == 'level3': if report_id == 'lvl3-failed-with-error': data = level3_service.get_level3_failed_with_error() elif report_id == 'lvl3-long-running-sessions': data = level3_service.get_level3_long_running() else: return JsonResponse({ 'error': f'Unknown report: {report_id}', 'data': [] }, status=404) elif app_id == 'mdm': if report_id == 'mdm-job-status': data = level3_service.get_mdm_job_status() else: return JsonResponse({ 'error': f'Unknown report: {report_id}', 'data': [] }, status=404) elif app_id == 'erp': if report_id == 'erp-job-status-latest': data = level3_service.get_erp_job_status() else: return JsonResponse({ 'error': f'Unknown report: {report_id}', 'data': [] }, status=404) else: return JsonResponse({ 'error': f'Unknown application: {app_id}', 'data': [] }, status=404) # Return successful response return JsonResponse({ 'data': data, 'source': 'oracle', 'count': len(data), 'timestamp': datetime.now().isoformat() }) except Exception as e: # Log error print(f"API Error [{app_id}/{report_id}]: {str(e)}") # Return error response return JsonResponse({ 'error': str(e), 'data': [], 'source': 'error' }, status=500)
Why This Design?
  • ✅ Single Endpoint: One URL pattern handles all reports
  • ✅ Dynamic Routing: Uses app_id and report_id to route requests
  • ✅ Centralized Error Handling: All errors caught and returned as JSON
  • ✅ Consistent Format: All responses have same structure
  • ✅ Easy to Extend: Adding new report just needs one elif block

11. Error Handling & Resilience

11.1 Error Handling Strategy

The portal implements multiple layers of error handling to ensure graceful degradation:

Layer 1: Database Connection Errors

# In oracle_client.py def fetch_all(query, params=None): connection = None try: connection = oracledb.connect(...) cursor = connection.cursor() cursor.execute(query, params) result = cursor.fetchall() return result except oracledb.DatabaseError as e: # Database-specific errors error_obj, = e.args print(f"Oracle Error {error_obj.code}: {error_obj.message}") raise except Exception as e: # General errors print(f"Unexpected error: {str(e)}") raise finally: # Always close connection if connection: try: connection.close() except: pass

Layer 2: Service Layer Errors

# In level3_service.py def get_level3_failed_with_error(): try: result = fetch_all(LEVEL3_FAILED_WITH_ERROR_QUERY) return result except Exception as e: print(f"Error in get_level3_failed_with_error: {str(e)}") # Return empty list instead of crashing return []

Layer 3: API Layer Errors

# In api_views.py def api_report_data(request, app_id, report_id): try: # Call service data = level3_service.get_level3_failed_with_error() # Return success response return JsonResponse({ 'data': data, 'source': 'oracle', 'count': len(data) }) except Exception as e: # Log error logger.error(f"API Error: {str(e)}") # Return error response (HTTP 500) return JsonResponse({ 'error': str(e), 'data': [], 'source': 'error' }, status=500)

Layer 4: Frontend Error Handling

// In JavaScript fetch(apiUrl) .then(response => { if (!response.ok) { throw new Error(`HTTP ${response.status}: ${response.statusText}`); } return response.json(); }) .then(data => { if (data.error) { showError(data.error); } else { renderTable(data.data); } }) .catch(error => { console.error('Fetch error:', error); showError(`Failed to load data: ${error.message}`); });
